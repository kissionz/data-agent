## 2. 产品基线

### 2.1 产品目标

InsightFlow ChatBI 面向企业经营和运营分析用户，通过自然语言完成指标查询、筛选、排序、趋势、同比环比、下钻与解释。大模型只承担理解、规划和解释：

1. 语义层裁决指标、维度、关系、版本和业务规则。
2. Planner 生成版本化 Analysis IR，不直接生成生产 SQL。
3. 确定性编译器把 IR 转为 SQL AST 和目标方言 SQL。
4. Query Gateway 负责权限注入、安全校验、预算、执行、缓存和取消。
5. Result Service 只允许答案引用结果集或已授权知识，并展示可核验依据。

### 2.2 核心成功指标（PRD 明确）

| 指标 | 目标 |
|---|---:|
| 用户获得答案、图表和上下文 | 30 秒内（总体体验目标） |
| 核心黄金问题执行准确率 | ≥ 95% |
| 认证指标语义准确率 | 100% |
| Intent Accuracy | ≥ 97% |
| Entity Linking F1 | ≥ 95% |
| Plan Accuracy | ≥ 93% |
| Groundedness | 数字 100%，陈述 ≥ 99% |
| Clarification Recall | ≥ 95% |
| Security Accuracy | 100% |
| 核心问答 API 可用性 | ≥ 99.9% / 月 |
| 首个状态反馈 P95 | ≤ 1.5 秒 |
| 缓存命中完整答案 P95 | ≤ 3 秒 |
| 常规查询完整答案 P95 | ≤ 15 秒 |
| 复杂分析完整答案 | ≤ 60 秒 |
| RTO / RPO | ≤ 30 分钟 / ≤ 5 分钟 |

### 2.3 范围边界

**MVP/P0**：可信模式；结构化数据；Web/API/嵌入式基础；查询、筛选、排序、趋势、同比环比、下钻、排行榜；身份租户、数据源、语义层、理解检索、规划澄清、编译执行、答案图表、管理后台、评测回放。

**V1/P1**：探索模式、高级分析、协作资产、开放 API 与 SDK；异常、归因、漏斗、留存等按功能总览逐步开放。

**明确非目标**：替代数仓或指标治理组织；通过对话修改生产数据；DDL/DML/任意脚本；对未治理或质量不合格数据承诺可信答案；以任意复杂 SQL 生成为成功目标。

## 3. 用户、角色与权限

### 3.1 角色能力矩阵

| 角色 | 核心任务 | 默认允许 | 默认禁止/限制 |
|---|---|---|---|
| 业务用户 | 查指标、趋势、筛选、下钻、导出 | 已授权业务域和行列范围内的可信查询 | 专家 SQL、语义发布、越域数据 |
| 业务负责人 | 经营复盘、跨部门对比、订阅 | 认证指标和授权汇总数据 | 未授权明细、平台配置 |
| 分析师 | 探索、归因、验证 SQL、沉淀案例 | 探索模式、受控只读 SQL、案例提审 | DDL/DML、未声明高风险 Join |
| 指标管理员 | 指标定义、认证、版本、下线 | 语义对象编辑、对账、审批/发布（按职责分离） | 默认读取业务明细 |
| 数据管理员 | 数据源、血缘、质量、枚举 | 连接、同步、目录与质量配置 | 模型运营、业务权限授予 |
| 安全管理员 | 策略、脱敏、审计、告警 | RBAC/ABAC、安全策略、审计检索 | 默认不代表用户查询业务数据 |
| 平台运维 | 模型路由、容量、SLO、灰度 | 平台配置、监控、回滚 | 默认不读取业务数据 |
| 服务账号 | API/嵌入式集成 | 授权作用域内的机器访问 | 交互登录、权限外数据 |

### 3.2 权限模型（PRD 明确）

- 隔离层级：租户 → 组织 → 工作空间 → 业务域。
- 授权：RBAC + ABAC；角色、用户属性和资源标签共同决策。
- 数据权限：业务域、行级、列级、脱敏、导出和数据分类策略。
- 身份继承：模型调用、检索、查询、缓存和导出全部继承真实用户身份。
- 防线：跨租户访问必须在 API 网关和数据层双重拒绝。
- 策略变更 5 分钟内生效；旧缓存不能绕过新策略。
- 分享接收者重新鉴权，不复用分享者的高权限结果。
- 运维与管理员权限不应隐式获得业务数据读取权。

### 3.3 建议补齐的权限细节

- 默认采用职责分离：指标编辑者不能单人完成同一版本的认证发布；安全策略编辑与审批分离。
- 建议定义资源动作：`view/query/export/share/manage/publish/approve/replay/audit`，而非只按页面授权。
- 权限拒绝统一返回不泄露资源存在性的文案；管理审计可记录真实拒绝原因，但需更高权限。
- 服务账号使用短期令牌、独立配额、可轮换密钥和明确 workspace/domain scope。

### 4.2 问答工作台详细规格

**输入区**

- 自然语言输入、发送、停止；问题为空时发送不可用。
- 明确展示当前工作空间、业务域、模式（可信/探索/专家）。
- 推荐示例只能来自当前业务域与可见语义对象，不能泄露无权资源名称。
- 流式展示阶段，而不是伪造逐字答案：已接收 → 理解中 → 需要澄清 / 正在校验 → 查询中 → 生成答案 → 完成。

**答案头部**

- 结论、可信/探索/专家标签、数据更新时间、语义版本。
- 显著显示默认条件、自动纠正、低新鲜度、部分结果和异常提示。
- 不以模型自报“高置信度”作为可信证据。

**结果区**

- KPI、图表、表格切换；图表类型必须经过字段类型、行数和误导性轴校验。
- 展示指标口径、时间范围、筛选条件、数据来源、新鲜度、结果引用。
- SQL 默认折叠；探索/专家模式更突出，可信模式可查看但不可任意执行编辑。
- 大结果分页；所有导出重新鉴权并应用水印、大小限制和脱敏。

**追问区**

- 结构化快捷操作：下钻、换维度、同比、环比、解释异常。
- 自然语言追问以结构化会话状态为执行事实源；显式条件优先，只有明确覆盖才替换旧条件。

### 4.3 响应式与可访问性（建议补齐）

- MVP 默认桌面优先，推荐设计基准宽度 1440px；最小可用宽度 1024px。移动端仅提供结果查看和轻量提问，不承诺完整管理后台。
- 键盘可完成提问、候选选择、停止运行、切换结果标签；焦点顺序和加载提示符合 WCAG 2.1 AA。
- 图表提供数据表替代视图、色盲安全配色和非颜色状态提示。
- 长数字按指标单位格式化，保留原始值下载；时区、自然周/财务周和币种必须明确。

## 5. 端到端用户流程

### 5.1 标准可信查询

1. 用户进入工作空间，选择业务域，默认可信模式。
2. 输入问题；前端生成幂等键并调用 `POST /v1/questions`。
3. 网关校验身份、租户、配额、请求策略，生成 request_id/trace_id。
4. 会话服务加载结构化状态；检索服务先按权限/业务域/版本过滤，再召回语义对象、术语、枚举和案例。
5. Planner 输出 Analysis IR；Schema、语义、兼容性、权限、歧义和预算依次校验。
6. 高置信且安全：直接执行并显示采用条件。中置信：执行但突出默认/纠正。低置信：进入澄清，不执行数据库查询。
7. 编译器生成 SQL AST 和方言 SQL；Query Gateway 执行白名单、权限注入、脱敏、成本/扫描预算、EXPLAIN 检查。
8. 查询执行；支持状态查询与取消，取消 3 秒内传递到底层引擎。
9. Result Service 做确定性派生、质量检查、分页和图表规格生成。
10. 回答模型只接收必要摘要，答案中的数字映射到结果单元格或确定性计算。
11. 页面展示结论、图表/表格、口径、筛选、时间、来源、新鲜度、语义版本和追问入口。
12. 全链路事件异步写入审计、监控与评测样本系统。

### 5.2 澄清流程

1. 触发条件：指标口径多义、关键时间歧义、成员值非唯一、Join 缺失/多义或其他关键实体不确定。
2. 系统返回 `AMBIGUOUS_QUERY` 与最多 3 个结构化候选；候选显示业务差异，不暴露无权对象。
3. 用户选择候选或补充问题；`POST /v1/runs/{id}/clarify` 必须绑定原 IR 和候选版本。
4. 服务端重新鉴权、检索和校验；不能仅信任前端提交的候选 ID。
5. 候选因语义发布或权限变化失效时，提示重新澄清，不静默选择。

### 5.3 多轮追问

1. 从当前结构化状态继承业务域、指标、维度、筛选、时间、粒度、排序、展示和语义版本。
2. 用户显式条件优先于默认；“换成周粒度”等明确指令只增量修改对应字段。
3. 跨业务域追问清空不兼容语义对象，重新检索并做权限检查。
4. 原语义版本已弃用时，历史回放使用原版本；新追问默认迁移到最新认证版本，并提示口径变化。

### 5.4 导出与分享

1. 导出前重新验证权限、数据分类、脱敏规则和大小预算。
2. 生成 CSV/XLSX/PNG/PDF；附带口径、时间、来源和水印；记录 `export.completed`。
3. 分享保存会话/问题引用，不把高权限结果副本直接授权给接收者。
4. 接收者打开时重新鉴权；若无权，显示通用拒绝或提供其权限范围内的重新运行选项。

### 5.5 语义对象发布

1. 管理员创建草稿，填写指标/维度必填字段与所有者。
2. 校验字段完整性、类型、粒度、Join Graph 和编译结果。
3. 运行参考 SQL 对账、黄金问题、权限、质量和性能测试。
4. 提交评审；展示版本 Diff、依赖影响和替代项。
5. 审批后认证发布；新请求使用最新认证版本，历史会话仍可回放原版本。
6. 弃用提供公告与迁移期；下线要求依赖清零或显式豁免。

## 6. 状态模型

### 6.1 Run 状态机（建议补齐为实现默认）

`CREATED → ACCEPTED → RETRIEVING → PLANNING → VALIDATING → CLARIFICATION_REQUIRED → VALIDATING → COMPILING → COST_CHECKING → QUEUED → QUERYING → PROCESSING_RESULT → ANSWERING → SUCCEEDED`

任一执行中状态可进入：

- `CANCELLING → CANCELLED`
- `FAILED`（不可恢复或重试耗尽）
- `PARTIAL_SUCCEEDED`（多步计划部分完成）
- `EXPIRED`（澄清或排队超时）

建议所有状态包含 `started_at/ended_at/progress_code/retryable/user_message/debug_reference`；用户文案与内部错误细节分离。

### 6.2 会话状态

- `ACTIVE`：可继续追问。
- `ARCHIVED`：默认不出现在主列表，可恢复。
- `DELETED`：建议采用软删除 + 租户留存策略，超过恢复窗口后物理清除。
- `READ_ONLY`：语义版本不可用、权限撤销或法律保留时仅允许有限查看。

### 6.3 语义对象生命周期

| 状态 | 可见/可用范围 | 进入条件 | 退出条件 |
|---|---|---|---|
| 草稿 | 管理员、指定测试用户 | 字段完整、可编译 | 提交评审或删除 |
| 评审 | 评审人、测试用户 | 对账任务创建 | 批准或驳回草稿 |
| 认证 | 可信模式用户 | 准确率、权限、质量、性能门禁通过 | 弃用 |
| 弃用 | 存量会话可用，新问题推荐替代 | 公告、替代方案、迁移期 | 下线或撤销弃用 |
| 下线 | 仅历史审计回放 | 依赖清零或显式豁免 | 不建议直接恢复；创建新版本 |

### 6.4 数据源状态（建议补齐）

`DRAFT → TESTING → GATE_FAILED / READY → ACTIVE → DEGRADED → DISABLED`。

元数据同步任务状态：`SCHEDULED/RUNNING/SUCCEEDED/PARTIAL/FAILED/CANCELLED`；Schema 变更状态：`DETECTED/PENDING_APPROVAL/APPROVED/APPLIED/ROLLED_BACK/REJECTED`。

### 6.5 反馈状态（建议补齐）

`NEW → TRIAGED → IN_REVIEW → RESOLVED / REJECTED → CANDIDATE_DATASET → GOLDEN_APPROVED`。线上样本必须先脱敏、去重、人工标注，不能直接进入黄金集。

## 7. 异常、空态与恢复策略

### 7.1 PRD 已定义错误

| 错误码 | 用户呈现 | 系统行为 | 推荐操作 |
|---|---|---|---|
| AMBIGUOUS_QUERY | 需要确认指标、时间或对象 | 返回候选，不执行查询 | 选择候选/补充条件 |
| SEMANTIC_NOT_FOUND | 当前业务域没有可用口径 | 推荐相近认证对象或联系管理员 | 换业务域/申请建模 |
| PERMISSION_DENIED | 无权访问该内容 | 不泄露存在性，记录安全事件 | 申请权限/返回 |
| QUERY_TOO_EXPENSIVE | 查询范围过大 | 阻断提交或终止 | 缩短时间/增加筛选 |
| DATA_STALE | 数据尚未更新 | 展示最近成功时间 | 取消或继续 |
| PARTIAL_RESULT | 部分步骤完成 | 展示完成内容与失败原因 | 重试失败步骤/缩小范围 |
| MODEL_UNAVAILABLE | 暂时无法理解复杂问题 | 降级为模板或结构化入口 | 重试/手动选指标维度 |

### 7.2 建议新增错误码

| 错误码 | 场景 | 默认行为 |
|---|---|---|
| AUTH_REQUIRED / SESSION_EXPIRED | 未登录或令牌过期 | 保留未发送输入；登录后重新提交，绝不自动重放导出 |
| RATE_LIMITED | 用户/租户/模型/查询配额触发 | 显示重试时间，不切换到越权资源池 |
| VALIDATION_FAILED | IR Schema、类型或兼容性失败 | 不执行 SQL，记录版本与字段路径 |
| UNSAFE_QUERY | DDL/DML、多语句、危险函数、Join 放大 | 阻断并产生安全/质量事件 |
| QUERY_TIMEOUT | 底层执行超时 | 取消数据库任务，允许缩小范围或稍后重试 |
| QUERY_ENGINE_UNAVAILABLE | 数据源/引擎不可用 | 熔断；优先合规缓存，否则失败 |
| CANCELLED_BY_USER | 用户停止 | 3 秒内向下传播，展示已完成阶段 |
| RESULT_EXPIRED | 结果超出留存期或数据版本失效 | 重新鉴权并重新运行 |
| EXPORT_TOO_LARGE | 导出超过行数/文件预算 | 建议加筛选或申请离线导出流程 |
| SEMANTIC_VERSION_CONFLICT | 编辑时版本已更新 | 展示 Diff，要求合并，不覆盖他人修改 |
| AUDIT_UNAVAILABLE | 审计链路异常 | 敏感查询/导出失败关闭；低风险进入受控缓冲 |

### 7.3 必备空态

- 新会话：示例问题、可查询业务域、模式说明。
- 无权限业务域：不显示不可见域名称，仅提示联系管理员。
- 空结果：明确“查询成功但无数据”，展示筛选/时间并提供放宽条件操作，禁止编造解释。
- 会话搜索无结果：清除筛选或创建新会话。
- 语义中心无对象：引导创建草稿或导入，但不把未认证对象用于可信模式。
- 运营图表暂无数据：区分采集延迟、筛选无样本和权限不可见。

## 8. 功能域验收标准汇总

### F01 身份、租户与工作空间

- OIDC/SAML 登录、登出、禁用、SCIM 同步均生成审计事件。
- 跨租户请求在网关和数据层双重拒绝。
- RBAC/ABAC 策略 5 分钟内生效，缓存不可绕过。
- 模型、检索、查询和导出链路能定位真实用户、代理服务与策略版本。

### F02 数据源与元数据目录

- 适配 PostgreSQL、MySQL、ClickHouse、StarRocks、Trino、Snowflake、BigQuery 的插件接口；新增方言不改变规划协议。
- 凭据托管密钥服务；未通过连通性、权限、方言、扫描预算和质量门禁的数据源不可进入可信模式。
- 增量发现元数据变更并支持审批、回滚；枚举采样遵循脱敏与分类标签。

### F03 语义层与指标中心

- 认证指标字段完整率 100%，变更有所有者、审批、不可变版本。
- 成员值映射唯一，否则澄清；未声明、多义或高风险 Join 被拒绝。
- 派生、同比环比、累计、窗口、半可加指标与参考 SQL 数值一致（浮点误差除外）。
- 下线前完成影响分析；历史会话可按原版本回放。

### F04 自然语言理解与检索

- 黄金集实体链接 F1 ≥ 0.95；中文口语、简称、错别字、英文缩写和日期表达通过率 ≥ 95%。
- 检索必须先做租户、权限、业务域和版本过滤。
- 查询改写不改变原约束；原问题与规范化表达都可审计回放。

### F05 分析规划与澄清

- Planner 只输出通过 JSON Schema 的版本化 IR；未知字段拒绝。
- 多步计划每步声明输入、输出、预算、依赖和终止条件。
- 指标口径、租户、时间或权限关键歧义不得静默默认。
- 步数、token、查询次数、扫描量、耗时超限安全终止并解释。

### F06 SQL 编译、验证与执行

- 相同语义版本和 IR 产生逻辑等价 SQL。
- DDL/DML、多语句、注释绕过、未授权函数在提交前阻断。
- 笛卡尔积、多对多放大、重复聚合、粒度冲突、除零在执行前发现。
- 用户取消 3 秒内传递到底层；复杂查询执行 EXPLAIN。
- 缓存键包含租户、权限摘要、语义版本、SQL 指纹和数据版本。

### F07 答案、图表与追问

- 所有数字映射到结果单元格或确定性计算。
- 图表规格通过类型、字段、行数和坐标轴校验。
- 用户无需查看 SQL 即可看到口径、筛选、时间、来源、新鲜度和语义版本。
- 追问增量修改结构化状态并保留显式约束。
- 空、部分、超时、低置信结果提供可行动反馈，不掩盖失败。

### F08 协作与资产沉淀

- 会话支持重命名、搜索、收藏、归档、权限分享。
- 分享接收者重新鉴权；导出有水印、大小限制、审计和脱敏。
- 已验证案例保存问题模板、IR、语义版本和适用范围，并经分析师审核。

### F09 管理后台与模型运营

- 模型路由/超时/温度/配额/降级链配置版本化，支持租户覆盖、灰度、回滚。
- Prompt、工具协议、IR Schema、检索策略版本变更触发兼容性与黄金集测试。
- 面板可按租户、业务域、模型和版本下钻准确率、澄清率、拒答率、延迟、成本与失败分布。

### F10 评测、反馈与回放

- 反馈关联完整链路且默认不暴露敏感数据。
- 黄金集覆盖标准、口语、歧义、多轮、越权、复杂 Join 和故障场景。
- 每次发布自动回归；P0 指标未达标禁止上线。
- 回放支持原版本与候选版本比较，包含脱敏问题、检索、IR、SQL、结果摘要和答案。

### F11/F12

原 PRD 没有详细条目和验收标准；在补齐前不能作为开发完成判断依据。建议见第 10 节。

## 9. 上线验收与测试矩阵

| 类别 | 必测内容 | 门槛 |
|---|---|---|
| 产品 E2E | 核心场景、澄清、多轮、空结果、取消、分享、导出 | P0 用例 100% 通过，无阻断缺陷 |
| 数据 | 指标参考 SQL 对账、时区边界、SCD、半可加、Join 放大、新鲜度 | 认证指标 100% 对账；告警有效 |

## 10. PRD 缺口与建议默认补齐

### 10.1 必须在开发前确认（高优先级）

| 缺口 | 风险 | 建议默认 |
|---|---|---|
| 首个业务域、30–100 个指标未确定 | 无法构建真实语义层和黄金集 | 选择指标成熟、高频、权限清晰的单一业务域；把业务负责人列为验收签字人 |
| 首批数据引擎未确定 | 连接器与方言范围失控 | MVP 只支持“主数仓 + 一个 OLAP”；其他适配器保留插件协议但不承诺生产级 |
| 语义层自研/采购未定 | 核心数据模型和管理 UI 无法稳定 | 先抽象 Semantic Service API；用项目内轻量版本仓库实现 MVP，避免绑定供应商 |
| 模型部署和供应商未定 | 合规、成本、延迟不可估 | 统一模型网关；供应商、模型、地域、留存均为租户级策略；生产数据不用于训练 |
| F11/F12 缺详细需求 | 高级分析、API/SDK 无可验收范围 | MVP 不实现 F11；F12 仅保留已有 `/v1` API，嵌入 SDK 延后 V1 |
| UI 规范只有区域级描述 | 容易出现页面、组件和交互分歧 | 以第 4 节 IA 为默认；问答工作台优先，管理后台按权限分模块 |
| Analysis IR 只有字段摘要，无完整 Schema | Planner/Compiler 无稳定契约 | 在编码前冻结 `schema_version=1` JSON Schema、枚举、兼容策略和示例集 |
| 租户/组织/工作空间/业务域关系未定义基数 | 权限和导航实现分歧 | 租户含多个组织；工作空间属于一个组织；工作空间可挂多个业务域；成员资格可继承但显式拒绝优先 |
| 数据留存期限未量化 | 合规和存储成本不可验收 | 租户可配置；默认问题/IR/SQL 指纹 180 天、结果摘要 30 天、原始结果 7 天、审计 365 天；敏感数据可更短 |

### 10.2 建议补齐的 F11 高级分析范围

V1 建议拆分为：

- F11-01 异常检测：只对已认证指标和明确时间序列运行；展示算法、基线、窗口和阈值。
- F11-02 贡献/归因：必须使用声明的维度候选和可加性规则；区分相关性与因果结论。
- F11-03 漏斗：步骤、窗口、去重主体和顺序必须显式建模。
- F11-04 留存：主体、cohort、回访事件、时间粒度必须显式配置。
- F11-05 预测：默认仅探索模式；展示训练截止时间、置信区间和误差，不作为认证事实。

验收默认：每类能力都有标准数据集、参考实现、边界条件、预算上限和“不可回答”用例；答案不得把统计相关写成因果。

### 10.3 建议补齐的 F12 API/SDK 范围

- API 认证：OIDC Client Credentials 或用户委托令牌；禁止长期静态万能密钥。
- 幂等：`POST /v1/questions` 必须支持租户范围幂等键与明确有效期。
- 流式协议：建议 SSE，事件包含 `run_id/event_id/type/timestamp/payload/schema_version`，支持断线续传。
- Webhook：签名、重放保护、退避重试、死信队列；Webhook 不携带超出订阅者权限的数据。
- 嵌入式 SDK：Host 负责获取短期 embed token；组件不能接触数据库凭据。
- 版本：`/v1` 向后兼容；破坏性变更启用新主版本并提供迁移期。

### 10.4 关键数据与接口缺口

- 缺少完整的 IR JSON Schema、错误响应 Schema、事件 payload Schema、分页游标和排序稳定性规则。
- 缺少各引擎的数据类型映射、时区/日历/财务周期、NULL、精度和大数处理规则。
- 缺少结果引用协议。建议每个答案引用使用 `result_id + row_key + column_id + transform_id`，避免依赖易变行号。
- 缺少缓存 TTL、数据版本生成和主动失效协议。建议可信查询优先事件驱动失效，TTL 仅作兜底。
- 缺少任务重试语义。建议模型/网络瞬时错误可重试；查询写入虽不存在，但昂贵查询重试前必须重新预算；用户取消后禁止自动重试。
- 缺少文件导出的行数/大小阈值。建议默认在线导出 100k 行或 50MB，以先到者为准；更大结果走受审计异步导出。

### 10.5 UX 与文案缺口

- 未定义页面路由、导航、断点、暗色模式、无障碍和国际化。建议 MVP 简体中文、桌面优先、WCAG 2.1 AA；所有时间与币种显式标注。
- 未定义流式阶段文案。建议展示事实阶段和耗时，不展示模型思维链。
- 未定义澄清候选卡片结构。建议显示“采用后将查询什么、与其他候选的差异、数据范围”，最多 3 项。
- 未定义 SQL 展示/编辑边界。建议可信模式只读；探索模式只读 + 复制；专家模式允许编辑但必须独立资源池、重新校验和审计。
- 未定义危险操作确认。建议撤销服务账号、发布/下线指标、批量导出、修改安全策略需二次确认；高风险策略变更需审批。

### 10.6 安全与治理缺口

- 未量化审计日志访问、脱敏字段和防篡改要求。建议审计 append-only、独立权限、完整性校验和访问再审计。
- 未定义 break-glass 紧急访问。建议工单审批、短时授权、明确目的、全量审计和事后复核。
- 未定义数据删除/法律保留冲突。建议法律保留优先，界面显示冻结原因和授权范围。
- 未定义 Prompt/案例/Embedding 删除传播。建议语义或权限撤销时同步撤销检索索引并使相关缓存失效。
- 未定义供应商故障时数据外发边界。建议降级模型必须满足同等地域、留存、训练禁用和数据分类策略，否则使用模板回答。

## 12. 开发就绪结论

原 PRD 对产品原则、架构边界、安全、SLO、评测和上线门禁描述较完整，足以支撑 MVP 架构与纵向切片立项；但尚不足以直接一次性开发全部功能。最关键的前置项是：确定首个业务域/引擎、冻结 Analysis IR v1、明确租户资源模型、确定模型合规边界、补充数据留存策略，并把 F11/F12 从愿景拆成可验收条目。

在未获得进一步决策时，可按本文件的“建议默认”启动 MVP：仅可信模式、单业务域、主数仓 + 一个 OLAP、桌面 Web、只读查询、版本化轻量语义层、完整审计与黄金集门禁；探索模式、高级分析、协作与嵌入式 SDK 延后到 V1。
